# Service Catalog Content Audit & Optimisation Plan

**Date:** 2026-06-14
**Scope:** TMaaS marketplace service content in Supabase (cards + PDP)
**Source of truth:** `src/data/services.ts` (186 non-bundle variants) → generators → Supabase
**Method:** Copy-editing skill (Seven Sweeps), focus on Sweeps 3 (So What), 4 (Prove It), 5 (Specificity)

---

## 1. How content flows to the DB

```
src/data/services.ts  (186 variants, the single source of truth)
        │
        ├─ scripts/generate-seed-sql.ts ──────────► supabase/seed-data.sql
        │     writes CARD fields: products.short_description, product_content
        │     (description, positioning), product_features, product_tags,
        │     product_timeline_milestones, audience, industry_relevance,
        │     business_impact, department
        │
        └─ scripts/generate-pdp-content-sql.ts ───► supabase/migrations/…seed_pdp_content.sql
              calls scripts/lib/pdpCopyGenerator.ts which pulls from:
                • src/lib/variantSeoCopy.ts   (hero description, positioning)
                • src/lib/serviceFaqsCopy.ts  (FAQs)
                • src/lib/whyItMattersCopy.ts (why-it-matters, scope)
              writes PDP fields: hero_summary, overview_paragraphs,
                audience_description, deliverables_summary, delivery_process,
                package_highlights, why_it_matters, faq_intro,
                variant_deliverables, product_faqs
```

**Key consequence:** Editing the DB directly is the wrong fix — it gets overwritten on the next `db:seed:sql` / `db:pdp-content:sql` run. The durable fix is to edit the **source** (`services.ts` + the copy libs), regenerate the SQL, and apply.

186 non-bundle variants = **31 solutions × 6 service types** (Assess, Design, AI Design, Deploy, AI Deploy, Managed).
Collections: operations 96, experience 60, ai 18, security 12.

| Field | Distinct values | Reality |

The 4th ("Architecture alignment for X") only changes per solution — identical across a solution's 6 variants. |
| `timelineMilestones` | **1 distinct** | Literally identical across all 186 variants: "Phase 1: Discovery & Alignment → … → Phase 4: Operational Handover". |

| `audience` | 31 | One per solution; same across all 6 variants. |

| `tags` | 84 | Formulaic: `[collection, serviceType, "transformation", "high-impact"]`. |

**Headline:** the most-visible card element — the feature list — is ~75% identical across the **entire** catalog and 100% identical within each solution's six cards. This is the primary cause of the "everything looks the same" effect.

## 3. Copy-quality findings (Seven Sweeps)

Beyond duplication, the templated copy fails several editing sweeps:

- **So What (Sweep 3):** features like "Phased delivery and milestone tracking" and "SLA-backed governance and reporting" are process/feature statements with no benefit bridge. They tell the buyer *what we do*, never *what they get*.
- **Specificity (Sweep 5):** pervasive vague verbs ("supports", "structured insight", "actionable detail", "drive outcomes"). No numbers, no concrete artefacts, no named tools/platforms. `timelineMilestones` is generic to the point of meaninglessness.
- **Prove It (Sweep 4):** zero evidence anywhere — no metrics, sample outputs, or named methods. (Acceptable for a prototype, but worth flagging.)
- **Voice (Sweep 2):** consistent and on-brand (British spelling, no em dashes, DigitalQatalyst team) — **keep this**; the `.cursor/rules/pdp-copy-style.mdc` voice rules are good and must be preserved.

## 4. Recommended fix (priority order)

**Principle:** make each field unique at the level it *should* vary, and rewrite for benefit + specificity. Not every field needs to be per-variant — but none should be catalog-wide identical.

### Tier 1 — Card fields (highest visibility, do first)
1. **`features`** → per **variant** (solution × service type). 4 features each, benefit-led, stage-appropriate (an Assess card and a Managed card for the same solution should list different things). Kill the 3 universal boilerplate lines.
2. **`timelineMilestones`** → per **service type** at minimum (Assess ≠ Deploy ≠ Managed), ideally tuned per solution. Currently 1 string for all 186.
3. **`description`** → keep unique but rewrite the per-service-type template clauses so siblings don't read identically; lead with the outcome for that specific solution+stage.

### Tier 2 — Per-solution card fields
4. **`businessImpact`**, **`audience`**, **`industryRelevance`** → already 1-per-solution; rewrite for specificity (concrete outcomes, named buyer roles). Optionally vary `businessImpact` by stage.
5. **`tags`** → add 1–2 solution-specific topical tags beyond the formulaic four.

### Tier 3 — PDP fields
6. **`variant_deliverables` descriptions** → rewrite the single shared sentence into per-deliverable, per-solution specifics.
7. **`package_highlights`** → vary per solution, not just per service type.
8. **`delivery_process` bodies** → inject solution-specific language into step bodies.
9. **`faqs`** → expand `EXACT_FAQ_OVERRIDES` / `SOLUTION_FAQ_FOCUS` coverage; rewrite generic answers for specificity.

### Constraints (must hold)
- Preserve voice rules: no em dashes; don't open hero with duration; British spelling; "DigitalQatalyst team"; name the solution not "digital presence".
- Don't touch IDs, slugs, prices, durations, `serviceType`, `collection`, popularity.
- After edits: `npm run db:seed:sql` + `npm run db:pdp-content:sql`, then apply SQL to Supabase.

## 5. Batching plan for subagents

Batch by **collection**, since solutions cluster naturally and a subagent can hold one collection's domain context:

| Batch | Collection | Solutions | Variants |
|-------|-----------|-----------|----------|
| A | experience | 10 | 60 |
| B1 | operations (part 1) | 8 | 48 |
| B2 | operations (part 2) | 8 | 48 |
| C | ai | 3 | 18 |
| D | security | 2 | 12 |

Each subagent receives: the variant list for its collection, the voice rules, the Tier 1–3 field spec, and worked examples. Output: edited `services.ts` entries (Tier 1–2) and/or copy-lib override entries (Tier 3) for its variants only — no shared-file races because each owns a disjoint variant-id range.

**Sequence:** Tier 1 across all batches first (fixes the visible card duplication), regenerate + review, then Tier 2, then Tier 3. Commit per batch.
