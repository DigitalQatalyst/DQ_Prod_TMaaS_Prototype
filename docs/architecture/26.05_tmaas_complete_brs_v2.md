# 2. Business Context

## 2.1 Organisation Context

TMaaS is positioned as a DigitalQatalyst (DQ) transformation operating platform designed to industrialise digital transformation delivery through a productised services model. The organisation operates within the digital transformation, advisory, technology delivery, and operational enablement space, serving organisations seeking structured transformation execution rather than isolated consulting engagements.

The platform reflects a shift from project-centric transformation delivery toward reusable operating models, standardised service blueprints, AI-assisted delivery coordination, and governed transformation execution models.

The operating model combines:

- Marketplace-led transformation discovery
- Structured transformation fulfilment
- Governance-led operational delivery
- Cross-functional stakeholder coordination
- AI-assisted transformation intelligence
- SaaS-enabled transformation activation

# 3. Platform Overview

## 3.1 Platform Purpose

TMaaS is a DQ-owned transformation operating platform and marketplace that enables organisations to discover, activate, govern, deliver, and scale digital transformation initiatives through standardised transformation services, AI-assisted operational intelligence, and structured delivery coordination across internal and external stakeholders.

TMaaS replaces fragmented transformation coordination environments built around spreadsheets, disconnected project tools, static PMO reporting, email-driven governance processes, and isolated operational trackers.

TMaaS complements rather than replaces delivery and operational tooling such as Azure DevOps, Microsoft Planner, spreadsheets, enterprise communication tools, finance coordination systems, and fragmented operational tracking environments. The platform sits above execution systems as the transformation orchestration, fulfilment, governance, operational coordination, and transformation intelligence layer across engagements.

The platform combines capabilities that are usually disconnected across enterprise ecosystems:

- Marketplace discovery
- SaaS-style service activation
- Operational transformation fulfilment
- Governance oversight
- Portfolio intelligence
- Commercial coordination
- AI-assisted transformation operations

No existing single platform combines transformation marketplace discovery with governed downstream operational fulfilment, portfolio governance, AI-assisted delivery intelligence, and commercial alignment within a unified operational architecture.

### Navigation Model

TMaaS uses a role-led navigation architecture.

The platform renders:

- Role-specific landing experiences
- Role-specific navigation structures
- Shared operational entities with segmented permissions
- Contextual intelligence based on operational role
- Cross-domain operational visibility based on governance access

---

### Target Application Architecture

| Tier | Technology | Rule |
|---|---|---|
| Client | React / Next.js / React Native | UI rendering only. No business logic in client applications. |
| Application | Express / BFF | Business logic, orchestration, validation, workflow execution, and access control. |
| Data | Data API (Supabase / PostgREST / Hasura) + PostgreSQL + Redis | All database access occurs through controlled APIs. No direct raw SQL access from application code. |

---

### Universal Request Anchor

`s2_account.requests` acts as the universal operational anchor for all Stage 2+ transactions.

Every:
- application
- engagement
- request
- operational workflow
- transformation activation
- fulfilment process
- governance review
- commercial process

must map back to a governed request object.

Stage 4 transformation orchestration capabilities extend the request lifecycle rather than replacing it.

---

### Dependency Rule

Dependencies flow downward only.

- Higher stages may reference lower stages.
- Lower stages may never reference higher stages.
- Shared operational entities must remain platform-wide.
- Domain modularity must not fragment shared operational records.

# 6. Design Constraints

| Constraint | Rationale |
|---|---|
| Single data model rule | Domain modularity must not fragment shared operational entities across transformation workflows. |
| Three-tier architecture | Business logic must remain in the application tier to ensure governance consistency and operational control. |
| Universal request anchor | All Stage 2+ operational workflows must anchor to `s2_account.requests` to maintain lifecycle consistency. |
| Dependency direction rule | Higher stages may reference lower stages; lower stages must never reference higher stages. |
| Immutable audit trail | All operational state changes must remain permanently traceable and non-destructible. |
| ADR requirement | Significant architectural decisions must be documented before implementation begins. |
| Governance-first permissions | Role-based operational access must enforce transformation governance separation. |
| AI advisory-first deployment | AI recommendations remain advisory-only during early platform releases. |
| SaaS-ready marketplace activation | Marketplace architecture must support governed self-service checkout and activation flows. |
| Cross-industry operational flexibility | Platform structures must support multiple industries without requiring domain fragmentation. |
| Future orchestration readiness | Service blueprints must evolve toward executable orchestration capability without requiring architectural rebuild. |
| ERP integration readiness | Commercial operations architecture must support future enterprise integration expansion. |
